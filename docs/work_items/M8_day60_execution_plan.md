# M8 Day-60 Execution Plan (Days 31-60)

Date: 2026-03-02
Window: Day 31 to Day 60
Parent: `M7_30_day_execution_master_plan.md`
Goal: Deliver research-backed, ephemeris-driven orbit physics with cross-engine validation and fail-closed signoff behavior.

## 1) Day-60 Mission

By Day 60, satellite pass geometry and derived key-rate predictions must be:
1. generated by real orbital providers (not envelope-only heuristics),
2. validated against a reference lane,
3. traceable and reproducible with explicit uncertainty bounds,
4. blocked from GO when provider trust or parity gates fail.

## 2) Non-Negotiable Rules (Inherited + Enforced)

1. No stubs in certification/release mode.
2. No hardcoded hidden physics constants.
3. No parity-free backend promotion.
4. Every result artifact must include provider identity, version, seed (if stochastic), and source hashes.
5. Unknown trust state or out-of-domain model => HOLD.

## 3) Deliverables Due by Day 60

1. Orbit provider interface and production provider:
   - `skyfield` provider enabled for real TLE/ephemeris pass synthesis.
2. Secondary provider for mission analysis:
   - `poliastro` lane for comparative/analysis runs.
3. High-fidelity reference validation lane:
   - `orekit` validation adapter (prefer sidecar/service model).
4. Cross-engine parity test suite:
   - pass start/end, peak elevation, slant-range traces, visibility windows.
5. Artifact provenance upgrade:
   - provider metadata and input hashes emitted in all satellite-chain outputs.
6. CI upgrades:
   - provider/parity jobs run in dedicated satellite workflow.
7. Physics validity documentation:
   - model validity domain, known failure modes, and uncertainty budget method.

## 4) Scope and Boundaries

In scope:
1. Orbit physics providerization and validation.
2. Schema and certificate metadata updates for provider provenance.
3. CI + tests required for trust promotion.

Out of scope for Day 60:
1. Full API v1 rollout.
2. Full multitenant auth program.
3. Broad protocol-engine federation (`cirq`/`pennylane`/`tfq`).

## 7) Day-60 Acceptance Gates

1. `provider_gate`:
   - Skyfield provider active and validated in production lane.
2. `parity_gate`:
   - Skyfield vs poliastro vs orekit deltas within thresholds.
3. `uncertainty_gate`:
   - Uncertainty budget complete and below allowed bounds for GO.
4. `repro_gate`:
   - Deterministic replay matches reference within tolerance.
5. `evidence_gate`:
   - Signed artifacts include provider/version/hash/inputs.

Fail any gate => HOLD.

## 8) Metrics for Day-60 Review

1. Orbit parity pass rate >= 99% on validation fixtures.
2. Provider metadata completeness = 100% in certificates.
3. Deterministic replay success >= 95%.
4. Zero certification runs using untrusted provider path.
5. CI parity lane median runtime within agreed budget.

## 9) Risks and Mitigations (Day-60 Specific)

1. Orekit integration complexity:
   - Mitigation: sidecar boundary, not direct core coupling.
2. Cross-engine model differences:
   - Mitigation: explicit tolerance tiers and uncertainty rollup.
3. TLE data quality drift:
   - Mitigation: snapshot fixtures + source hash pinning.
4. Performance regression from high-fidelity providers:
   - Mitigation: tiered modes (`preview`, `certification`) and cached intermediates.
5. False confidence from passing unit tests only:
   - Mitigation: mandatory parity integration tests on CI.
